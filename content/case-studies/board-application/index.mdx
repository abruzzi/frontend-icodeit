This guide walks through a **kanban-style board** (columns like *Backlog* / *In progress* / *Done*, as in Trello or Jira): what to store, how to shape data and APIs, and where things break in production—written for **depth**, not a timed interview.

## What is a board application?

Teams use a **shared surface** where work items are **cards** grouped into **columns**. The main interaction is **moving a card** (reorder within a column or move across columns). Often, **several people** use the same board; sometimes columns are **huge** or **customized**.

Your implementation should stay **correct** when:

- the network is slow or flaky;
- the same move is **retried**;
- **two people** move cards close together in time;
- users might rely on **keyboard** for some interactions, not only drag-and-drop.

## How this guide walks it

There is no single architecture that is “best” for every product and team. **Here I use one path I like because it is easy to understand and to follow**—especially on the web: thin UI, clear data shape, then persistence—without pretending it is the only respectable approach.

Complicated systems are easier if you do not start from “everything at once.” The same idea is often called a **[steel thread](https://www.rubick.com/steel-threads/)**: the **thinnest slice that still threads through the real boundaries of the system** for one important use case, shipped so it holds in production—then you grow from that spine. Jade Rubick’s article is the clearest write-up I know of the term.

This walkthrough **stages** where that thread crosses the next boundary—it is the same iterative discipline, not a different philosophy. **Steps 1–2** are a **client-local steel thread**: faithful UI plus real move semantics in memory, so drag boundaries and ordering updates fail first without the network seam. **Step 3** threads the same use case **through persistence** (single-player, reload matches truth). You are only choosing to pierce **client ↔ server** once the in-browser spine is boring.

**1. Fixtures first, render correctly.** Start from **fixture data** (for example a JSON snapshot in the frontend, or assume you can always fetch it) and render **faithfully**: columns, card order, fields the UI needs.

**2. Interaction before the network.** Add **moving cards** and **order in client state only** so you learn the data updates and drag/drop boundaries without distributed-system noise. **Try it first** above is exactly **steps 1–2**.

**3. Persistence, single-player.** Save to the server next, **one user at a time**: a move persists and reloads correctly; **defer** notifications, realtime, and conflict UX until that baseline is boring.

**After that** (when you need them): **ack / rollback**, **idempotent** operations, **reload matches server**, **two clients** converging, WIP limits, automation, and the rest.

The rest of the article lines up with the **CCDAO** lens—**C**ollect constraints, **C**omponent structure, **D**ata modeling, **A**PI design, **O**ptimization—so each block maps to how you might walk the problem in a design review. **Data modeling** and **API design** go deepest on **step 3** and beyond.

## Product and engineering constraints

These decisions change your data model and API more than any library choice:

- **Columns**: fixed workflow vs user-defined; max count; WIP limits; “done” / archive rules.
- **Scale**: rough upper bound for cards per column (dozens vs thousands)—drives **pagination** inside a column vs load-all.
- **Permissions**: who may move which cards; read-only viewers.
- **Collaboration**: must moves appear for others within ~100ms, ~1s, or is **refresh** enough?
- **Accessibility**: is **full parity** without a pointer required?
- **Offline**: if moves queue locally, you need an explicit **merge** story when the client comes back.

## Why boards are tricky

- **Order is state.** Unlike a simple CRUD form, the product *is* the ordering of ids inside each column. Small bugs look like “my card jumped” or “it duplicated.”
- **Moves are concurrent.** Two users can target the **same index** or the **same card**; the server must be the **authority** on the final order (or you expose conflicts explicitly).
- **Optimistic UI is the default expectation.** Drag-and-drop feels broken if every move waits for the network; that means **pending state**, **rollback**, and **idempotent** server handling.

## UI structure and where state should live

Typical regions:

- **Board shell**: loads board metadata, permissions, and the **initial snapshot** (or starts subscriptions).
- **Column list**: renders columns in order; each column owns **only presentation** for that lane (title, WIP badge, drop target).
- **Cards**: mostly **presentational**; read from shared data; emit **intent** (“move card X after Y in column Z”) rather than mutating a global blob from deep inside a leaf component.
- **Drag / drop or keyboard layer**: keeps **ephemeral** state (what is being dragged, hover targets) **separate** from the **server-backed** board model. Mixing them in one object is a common source of impossible-to-replay bugs.

**Rule of thumb:** the **canonical board order** lives in one place (normalized store + ordered id lists). Gestures **propose** updates; the store **applies** confirmed or optimistic patches.

The **Try it first** demo uses the same idea at the leaf: columns own drop targets; cards are **presentational** and read shared column data; drag-and-drop **proposes** a new assignment (there, only client state; in production, an operation + ack).

## Data modeling (deep dive)

### Entities and relationships

At minimum:

| Entity   | Role |
| -------- | ---- |
| `Board`  | Id, title, workspace, settings (e.g. column ids in order). |
| `Column` | Id, board id, title, **order** among columns. |
| `Card`   | Id, board id, **column id**, **position** (index or sort key), title and other fields. |

Cardinality: a board has **many** columns; a column has **many** cards. Cards belong to **one** column at a time in the classic model.

### Server canonical model vs client view

The **server** stores the **source of truth** for column order and each card’s `(columnId, position)` (or equivalent). The **client** usually mirrors that in a shape optimized for rendering:

- **`cardsById: Record<CardId, Card>`** — full card records.
- **`columnId → orderedCardIds: CardId[]`** — render order per column.

Why both? The list gives you **order** in O(n) render without scanning all cards; the map gives you **O(1)** lookup when a realtime event references `cardId` only.

### Ordering: index vs sort key

Two common patterns:

1. **Integer index** per column (`position: 0..n-1`). Simple to reason about; **every insert in the middle** can force **many rows** to update on the server unless you use fractional indexing or another trick.
2. **Lexicographic / fractional rank** string between neighbors (`rank` between `"a"` and `"b"`). Fewer row updates on insert; slightly more complexity in the client when computing “between.”

For learning, **explicit index + server normalizes** is enough; optimize when profiling shows write amplification.

### LexoRank: easy to hand-wave until it hurts

**LexoRank** is Atlassian’s open, battle-tested take on fractional / lexicographic ordering—Jira boards are the usual reference implementation people point to. Conceptually it is still “a string sort key between neighbors,” but the **details** (how you bucket, rebalance when strings get too long, and keep inserts cheap under concurrency) are where toy implementations diverge from production.

Teams often **start with integer `position`**, ship, and only later notice **write amplification** (many rows touched on one insert), **hot spots** in the index, or painful **migrations** when two clients resolve “move after card X” differently under load. LexoRank is the pattern many mature products adopt **before** those fires; it is also easy to **skip in interviews and greenfield code** because “just use an index” sounds sufficient until scale and collaboration show up.

For a walkthrough aimed at implementers, see [*Reordering Lists at Scale (Without Breaking Everything)*](https://www.youtube.com/watch?v=Z6c7txCqsu8) on YouTube. Atlassian publishes a reference implementation in the [**LexoRank** repo](https://bitbucket.org/atlassian/lexorank/src/main/). Roadmap: a focused **LexoRank for card ordering** pattern lives on the [Patterns](/patterns) index; this board guide stays the scenario-shaped home for *when* ordering strategy matters in a kanban model.

### Moves as operations (recommended mental model)

Treat a move not as “set these three fields” but as an **operation** the server can **validate**, **order**, and **dedupe**:

```text
{
  "operationId": "uuid-client-generated",
  "type": "move_card",
  "cardId": "c_1",
  "fromColumnId": "col_a",
  "toColumnId": "col_b",
  "afterCardId": "c_9"   // or toIndex: 3 — pick one convention and stick to it
}
```

- **`operationId`**: same id on **retries** so the server does not apply the same move twice.
- **`afterCardId` (or index)** defines **intent**; the server may **adjust** if the neighbor moved (return final state).

The client keeps a **pending operations** list or marks cards as **pending** until ack.

### Concurrency: version or sequence

Every board snapshot should carry something the client can compare:

- **`version`** (monotonic integer) on the board or per column, or
- a **`seq`** stream for **applied operations** (see API below).

On conflict, either:

- **Server wins**: respond with **409** + **latest snapshot** or delta, client rolls back optimistic UI; or
- **Merge**: rare for whole-card moves; more relevant if multiple fields edit the same card simultaneously.

Clients should track **`lastSeenVersion`** or **`lastAppliedSeq`** and ignore **stale** events.

### Hot paths

- **Initial load**: one round-trip for board + columns + cards (or paginated per column).
- **Move**: one mutation + optional **broadcast** to other tabs/users.
- **Search within board** (if any): usually a **separate** read model or query, not a full scan of the client cache.

## API design (deep dive)

### Transports

- **HTTP** for **snapshot load** and **mutations** is standard.
- **WebSocket** (or **SSE**) when others must see moves **without** polling: server pushes **operation applied** or **patched state** after each accepted move.

You can ship v1 with **HTTP only** and periodic refresh; collaboration then adds the second channel.

### Read: board snapshot

A practical first contract:

`GET /boards/:boardId`

Response sketch:

```json
{
  "board": { "id": "b1", "title": "Sprint 12", "version": 42 },
  "columns": [
    { "id": "c_todo", "title": "To do", "order": 0 },
    { "id": "c_doing", "title": "In progress", "order": 1 }
  ],
  "cards": [
    { "id": "card_1", "columnId": "c_todo", "position": 0, "title": "…" }
  ]
}
```

If a column can hold **thousands** of cards, split:

`GET /boards/:boardId/columns/:columnId/cards?cursor=&limit=`

and document **how cursor behaves** when new cards are inserted at the top.

### Write: submit move (idempotent)

`POST /boards/:boardId/operations`

Headers: `Idempotency-Key: <operationId>` (or put `operationId` in the body and enforce uniqueness server-side).

**Success** returns:

- updated **`version`** (or next **`seq`**), and either
- **full snapshot** (simplest for small boards), or
- **minimal patch** (`{ affectedColumnIds, cards, order }`) to merge client-side.

**Reject** (validation, permission) returns **4xx** with a body the UI can show; client **rolls back** optimistic state.

**Conflict** (`409`): body includes **server truth** (snapshot fragment or “reapply from version 43”) so the client can reconcile without a full mystery refresh.

### Realtime: event stream (optional)

After collaboration ships, a minimal event shape:

```json
{
  "seq": 1001,
  "type": "move_applied",
  "payload": { "operationId": "…", "…final positions…" }
}
```

Rules that save you later:

- Events are **strictly ordered** by `seq`.
- On reconnect: **`GET …?afterSeq=lastSeen`** or **resync snapshot** if gap detected.
- **Ignore** unknown `operationId` duplicates—idempotency applies on the client too.

### Auth

- Attach **workspace / user** context on every call.
- Prefer **403** for “no permission” vs **404** for “board does not exist” based on your privacy product policy—be consistent.

## Performance, reliability, and accessibility

- **Drag performance**: prefer **transform**-based dragging and avoid **layout thrash**; virtualize columns only when profiling proves you need it.
- **Realtime bursts**: **batch** UI updates per animation frame; dedupe by `operationId`.
- **Keyboard**: explicit **move to column** / **move up-down** commands; **roving tabindex** inside the board; **announce** column changes for screen readers.
- **Retries**: exponential backoff on **5xx** / network; never assign a **new** `operationId` for the **same user gesture**.

## Trade-offs (when you evolve past the first slice)

| Tension | One side | Other side | When the “other” wins |
| ------- | -------- | ---------- | --------------------- |
| Conflict handling | Server-normalized order after each move | Operational transform on shared fields | Multiple users edit **the same card body** concurrently |
| Transport | Poll + refetch | WebSocket push | Sub-second multi-user updates |
| Optimism | Wait for server | Optimistic + rollback | Drag-and-drop UX expectations |
| Column size | Load all cards | Paginate in column | Very large backlogs |

## Common failure modes

- **Duplicate cards** after a retry → **idempotency** on `operationId` + server dedupe.
- **Corrupt order** after tab sleep → **resync** from `version` / `seq`, not patch stacking forever.
- **Ghost moves** (UI shows move, server never got it) → **pending** timeout + user-visible retry or revert.
- **Accessibility gaps** → treat keyboard and SR as **v1**, not a later pass.
