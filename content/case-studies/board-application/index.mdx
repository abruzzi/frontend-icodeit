This guide walks through a <Highlight variant="underline3">kanban-style board</Highlight> application: work items are **cards**, grouped into **columns** like *Backlog* / *In progress* / *Done* (as in Trello or Jira). We'll discuss common challenges, patterns, and strategies that address them—and the trade-offs behind those choices.

We'll focus mainly on **frontend concerns**: how to manage state, how to shape data and APIs, and where things break in production—written for **depth**. You can use it to prepare for a system design interview, but it also helps you develop a feel for how systems are designed and how to make decisions when you have many options.

## Introducing board application

In practice, a board is a **shared place** where work is represented as **cards** in **columns**. People create and edit cards, and they **move** them—reordering within a lane or jumping across columns. When **several people** share a board, you also have to think about concurrent edits, how moves propagate, and what “correct order” means when two gestures land close together.

Products like Jira or Trello made this shape familiar; the ideas transfer, but the **hard parts** are rarely the rectangle layout. They show up in **ordering**, retries, collaboration, and the gap between what the UI shows and what the server will accept.

Let's **start with something we can touch.** The widget below shows **columns** and **ordered cards** loaded from a JSON-shaped fixture—drag cards between *Backlog*, *In progress*, and *Done*. It uses [Pragmatic drag and drop](https://atlassian.design/components/pragmatic-drag-and-drop/) and **local state only** (no server), so you see the core UX before we talk about APIs and persistence.

<Callout tone="note">
This demo does not call an API. Later sections cover idempotent **move** operations and reconciling on ack or rollback—see **API design** and **Data modeling**.
</Callout>

From here on, it helps to split what you're building into two buckets: **functionality** (the features users see) and **cross-functional** requirements (how the system behaves under pressure).

Functionality is the main feature of the application. For our case, we need to render a board, let users create and edit cards, and move cards across columns. We also need to support **multiple users** working on the same board at the same time.

Cross-functional requirements include reliability, performance (both actual speed and perceived performance), security, accessibility, internationalization, and more. They’re equally important as functionality, because they determine whether the product feels solid in the real world.

For that, your implementation should stay **correct** when the network is slow or flaky (**reliability**), when the same move is **retried** (maybe by another user), when **two people** move cards close together in time, and when users rely on the **keyboard** for some interactions instead of only drag-and-drop (**accessibility**).

If you’re preparing for a system design interview, it helps to practice **clarifying questions** that narrow scope before you draw boxes: what you must ship first, how big the product can get, and which non-functional bar you’re held to. That framing makes the strategies in later sections easier to pick—you’re choosing for a stated problem, not a generic “board app.”

<Callout tone="essentials">
Interview-style clarifying questions often look like this:

- **MVP vs later**: Beyond columns and cards, do we need search, filters, attachments, or custom fields—or can we treat those as out of scope for v1?
- **User and permissions**: Is this single-team internal tooling or multi-tenant? Who can create boards, edit cards, or move cards—any viewer vs editor vs admin?
- **Devices and offline**: Primary surface desktop web, mobile web, or native? Do we need offline edits (queued moves) or is online-only acceptable?
- **Collaboration bar**: Single user for v1, or realtime multi-user from day one? If multi-user, is “refresh to see updates” enough, or do we need sub-second visibility?
- **Scale**: Order of magnitude for boards, columns, cards per column, and concurrent editors—enough to rule out “load everything into the client”?
- **Ordering and conflicts**: Is strict ordering per column non-negotiable? What should happen when two users move or reorder the same card at once—server wins, last-write-wins, or explicit conflicts?
- **Performance and UX**: What feels “fast” (e.g. drag latency, time-to-interactive after load)? Any SLA for mutations or for seeing someone else’s move?
- **Accessibility**: Is keyboard-only parity required for moving cards, or is pointer DnD the primary path for v1?

You don’t need every answer up front—but **one sentence each** on scope, collaboration, and scale usually saves you from the wrong architecture.
</Callout>

## How this guide walks it

There is no single architecture that is “best” for every product and team. **Here I use one path I like because it is easy to understand and to follow**—especially on the web: thin UI, clear data shape, then persistence—without pretending it is the only respectable approach.

Complicated systems are easier if you do not start from “everything at once.” The same idea is often called a **[steel thread](https://www.rubick.com/steel-threads/)**: the **<Highlight variant="underline2">thinnest slice</Highlight> that still threads through the real boundaries of the system** for one important use case, shipped so it holds in production—then you grow from that spine. Jade Rubick’s article is the clearest write-up I know of the term.

For this board application, a reasonable split can be as follows:

1. Start from fixture data (for example a JSON snapshot in the frontend, or assume you can always fetch it), and render the board faithfully—columns, card order, and the fields the UI needs.

2. Add moving cards with ordering kept in client state only, so you learn how updates and drag-and-drop boundaries behave before you layer on distributed-system concerns.

3. Persist moves to the server next, one user at a time, until reload shows the same board—and defer notifications, realtime, and conflict UX until that baseline is boring.


The above approach is simple and focuses on the frontend first. You can work safely on the UI (design + code) and <Highlight variant="underline3">iterate quickly</Highlight>. The benefit is you always have something to show a product owner or user, so you can get early feedback and adjust accordingly. You can design with mock data—avatars, default fallbacks, empty states—assuming the backend will eventually return the shape the UI consumes.

The drawback, on the other hand, is you’re delaying integration, which can bite you later. If the backend can’t handle a case efficiently (or the API contract can’t support the UX you designed), you might have wasted time on UI work. You can also make assumptions that you won’t realize are wrong until much later.

Another split can be (which is usually my favorite if I have to implement it):

1. Add a card (only with a title), and call a backend service (the backend can decide to use a mock or actually persist to a database)
2. Refresh the board and you should now see the new card
3. Edit the card details (e.g. change the title), and call `update` endpoint (could be updating the in-memory array in the backend)
4. See the changes applied
5. And implement more end-to-end integrations (realtime updates, multi-user collaboration, etc.)

The second approach has the same principle as the first one: we delay architecture decisions until we have to. The reason is simple—those decisions are normally *hard to reverse*. If you start with a non-SQL database, ship with some data already inside it, and then later realize a relational database is a better fit, you end up dealing with downtime and migration work.

So try to delay the decision until later, when you have a better understanding of what you're **building**, what the constraints are, and what options you have—so you can make better **decisions**.

Again, there isn't a right or wrong split here—but the goal stays the same:

**After that** (when you need them): **ack / rollback**, **idempotent** operations, **reload that matches the server**, **two clients** converging, WIP limits, automation, and the rest.

The rest of the article lines up with the [**CCDAO**](https://www.youtube.com/watch?v=ojFsLG4TkUA) lens—**C**ollect constraints, **C**omponent structure, **D**ata modeling, **A**PI design, **O**ptimization—so each block maps to how you might walk the problem in a design review. **Data modeling** and **API design** go deepest from **step 3** onward.
</Callout>

## Designing the UI

For most frontend engineers, UI is the easiest place to start because it’s concrete and close to what we do every day. So let’s begin with the board page and sketch the **happy path** first: the board has a few columns, each column already has some cards, and the initial fetch succeeds.

From that snapshot, a reasonable first component list looks like:

| Component   | Role |
| ----------- | ---- |
| Board       | Name, meta (e.g. total card count), and the overall layout. |
| Filters     | Status, assignee, labels, etc. (if in scope). |
| Search      | Query input + results behavior (if in scope). |
| Column      | Title, card count / WIP badge, drop target. |
| Card        | Title + lightweight metadata (assignee, labels, due date). |
| UserAvatar  | Consistent avatar + fallback rendering. |

But that list is still *read-only*. Once you ask “where does this data come from?” the UI immediately grows:

- Can users add/rename/reorder columns?
- Are filters built-in, or can users create saved views?
- When a card is edited, is it inline, a side panel, or a modal?

And then there’s the part people forget when they only design the happy path.

### Loading

Initially, when the page starts loading, it’s safe to assume not every user has perfect network conditions. They might be on 4G, travelling, or switching networks—so a blank page is rarely the right experience. Design for this state explicitly: use a spinner or a skeleton to make it feel *fast* while you fetch the snapshot.

## Data modeling (deep dive)

Once we have a solid picture of the UI—what we are including, how it loads, and which edge states we actually own—we can reason much more clearly about <Highlight variant="underline3">data modeling</Highlight>. The question becomes: what shape stays correct when cards move, assignees change, and events arrive out of order?

It’s common to first render from a nested snapshot (often what a read API returns), then introduce a <Highlight variant="underline4">normalized client shape</Highlight> when duplication and partial updates become painful.

### Entities and relationships

At minimum:

| Entity   | Role |
| -------- | ---- |
| `Board`  | Id, title, settings (e.g. column ids in order). |
| `Column` | Id, board id, title, **order** among columns. |
| `Card`   | Id, board id, **column id**, **position** (index or sort key), title and other fields. |
| `User`   | Id, display name, **avatar** URL, etc.; cards reference an assignee by **user id** (or null). |

Cardinality: a board has **many** columns; a column has **many** cards. In the classic model each card belongs to **one** column at a time. A user may be the assignee on **many** cards; each card has at most **one** assignee in the usual model.

<BoardDataModelDiagram />

<Callout tone="pitfall">
This is a **frontend-facing** description, not a mandate for how your database must look. The UI still needs **derived** values—total cards on a board, WIP against a limit, light aggregations—that might come from the API as precomputed fields, from a quick client pass, or from a mix of both. What matters on the client is knowing **where to patch** when something changes, so you are not scanning deeply nested trees for every small edit.
</Callout>

### Nested snapshot

The following is a typical response from the **server side**: columns at the top level, each column embedding its `cards` array with the fields the UI needs.

```json
{
  "id": "board-1",
  "name": "Board 1",
  "cardCount": 3,
  "columnCount": 3,
  "columnIds": ["col-1", "col-2", "col-3"],
  "columns": [
    {
      "id": "col-1",
      "title": "Backlog",
      "cards": [
        { "id": "card-1", "title": "Setup project structure" },
        {
          "id": "card-3",
          "title": "Configure ESLint & Prettier",
          "assignee": {
            "id": 2,
            "name": "Charlie Moore",
            "avatar_url": "https://i.pravatar.cc/150?img=2"
          }
        }
      ]
    },
    {
      "id": "col-2",
      "title": "In Progress",
      "cards": [
        {
          "id": "card-2",
          "title": "Install dependencies",
          "assignee": {
            "id": 2,
            "name": "Charlie Moore",
            "avatar_url": "https://i.pravatar.cc/150?img=2"
          }
        }
      ]
    },
    {
      "id": "col-3",
      "title": "Done",
      "cards": []
    }
  ]
}
```

Such a data structure is easy to use on the frontend, because we can render components by **traversing** the tree in a straightforward way: walk the `columns` array, and for each column walk its `cards`. The payload lines up naturally with a `Board → Column → Card` component tree.

```jsx
function Board({ data }) {
  return data.columns.map((col) => <Column key={col.id} data={col} />);
}

function Column({ data }) {
  return data.cards.map((card) => <Card key={card.id} data={card} />);
}

function Card({ data }) {
  return (
    <article>
      <h2>{data.title}</h2>
    </article>
  );
}
```

On the other hand, a nested structure like this makes it harder to update a **particular** piece of data when the same entity is **copied** in more than one place. In the example above, both **card-2** and **card-3** are assigned to **Charlie Moore**, and Charlie’s profile (including `avatar_url`) appears twice in the tree. If the user changes their avatar on a profile screen, how do we bring the board back in sync? We can **refresh the whole board** and re-render everything, or we can **search the tree** and apply a precise update to every node that embeds that user—but either approach takes extra effort, and as the board grows it is easy to miss a nested copy or to do redundant work.

### Normalized client state

A better approach is to **flatten** the data into a format where we only need to update **one place** for a given entity, and every usage of that entity is **derived** from that single source of truth. References between entities are **ids**, not duplicated objects. In a database or in traditional backend design this is usually called **normalisation**; on the frontend you will see the same idea in write-ups about **normalized state** and client caches (for example Redux-style stores). A **normalised** snapshot that carries the same board might look like this:

```json
{
  "id": "board-1",
  "name": "Board 1",
  "cardCount": 3,
  "columnCount": 3,
  "columnIds": ["col-1", "col-2", "col-3"],
  "columns": {
    "col-1": {
      "id": "col-1",
      "title": "Backlog",
      "cards": ["card-1", "card-3"]
    },
    "col-2": {
      "id": "col-2",
      "title": "In Progress",
      "cards": ["card-2"]
    },
    "col-3": {
      "id": "col-3",
      "title": "Done",
      "cards": []
    }
  },
  "cards": {
    "card-1": {
      "id": "card-1",
      "title": "Setup project structure",
      "assignee": null
    },
    "card-2": {
      "id": "card-2",
      "title": "Install dependencies",
      "assignee": 2
    },
    "card-3": {
      "id": "card-3",
      "title": "Configure ESLint & Prettier",
      "assignee": 2
    }
  },
  "users": {
    "2": {
      "id": "2",
      "name": "Charlie Moore",
      "avatar_url": "https://i.pravatar.cc/150?img=2"
    }
  }
}
```

<BoardNormalisationStepperDemo />

<Callout tone="essentials">
Note how each kind of object has **its own** top-level section—`columns`, `cards`, and `users` (or `assignees`). Each node holds the fields that belong to that entity, and when it needs another entity it **references** it through an `id` instead of embedding a second full copy.

There is no duplicated user payload in this normalised shape: when we need to update the avatar for user `2`, we open the `users` section, update that one record, and we do **not** have to touch the `cards` entries except that they already point at the same id.
</Callout>

### Server canonical model vs client view

Even when the client keeps a **normalised** map for rendering, **persistence** still belongs to the **server**. The server decides what is actually stored: which column a card lives in, in what **order**, and what happens when two moves race. The **client** is a mirror—it applies patches, shows optimistic UI, and must be able to reconcile when the server says “here is the truth now.”

The normalised JSON above already hints at two **different jobs** hiding inside “show the board”:

1. **Look up a card by id** — In the example, every full card body lives under `cards`, keyed by id (`"card-1"`, `"card-2"`, …). When a realtime payload or a mutation response says “something changed on **card-3**,” you jump straight to `cards["card-3"]` and update **one object**. You do **not** hunt through `col-1`, then `col-2`, to find which nested array still holds that card. In TypeScript people often write that table as **`Record<CardId, Card>`** or `cardsById`.

2. **Know the order inside each column** — In the same example, `columns["col-1"].cards` is **not** an array of full card objects anymore; it is `["card-1", "card-3"]`: an **ordered list of ids** for *Backlog*. To paint that column in the UI, you walk that list in order, and for each id you read the title and assignee from `cards[id]`. That list is your **source of truth for ordering** on the client (until the server sends a correction). People often name that structure **`columnId → orderedCardId[]`**.

Why keep **both** instead of only the map? If you only had `cards` and each card carried `columnId`, you could still render, but every frame you would be asking “**which** cards belong to *Backlog*, and **in what order**?”—usually by scanning **all** cards. The per-column id list answers that in one pass: you render **at most** the cards that appear in `["card-1", "card-3"]` for *Backlog*, in exactly that sequence, without touching **card-2** in *In Progress*. When an event names a single `cardId`, the map still lets you jump to the record in **one step** and update assignee, title, or flags without walking the whole board.

So: **server** owns the canonical order and persistence; **client** holds a **table of cards by id** plus, for each `col-*`, an **ordered list of those ids**—the same split you already see in the normalised fixture, just spelled out as *why* it is there.

## Desiging the API

At some point the board has to talk to a **real backend**, and that is when API design stops being a tidy diagram and turns into a **conversation**. I have never seen it work well as a solo act—backend and frontend both have constraints worth listening to—but I still think the **shape the UI actually consumes** (nested vs normalised, field names, how you represent ordering) ought to win when there is tension, because that is what you render and patch every day. We already sketched that side in **Data modeling**; the API should line up with it instead of fighting it.

Traditionally—and I mean the ordinary, mainstream history of our industry, not an academic detour—most teams still reach for **REST-flavoured HTTP + JSON**: `GET` for reads, `POST` or `PATCH` for writes, routes that read like resources. Every ecosystem has a DSL that makes the first draft look easy.

```js
app.get("/boards/:boardId", () => {
  /* return snapshot */
});

app.post("/boards/:boardId/cards", () => {
  /* create card */
});
```

What I like to do next is take the operations you already named in **Collect constraints**—what must exist for v1, how big things can get, whether multiple people are in the same board at once—and **map them straight onto endpoints** while you are still allowed to be boring. The interesting work is not the first happy-path sketch; it is what happens when **retries**, **concurrency**, and a **suddenly enormous backlog** show up and your innocent `GET` is no longer innocent.

### Transports

**HTTP** is still the spine: load the snapshot, submit mutations, stare at status codes in DevTools without needing a second mental model. When the product truly needs other people to **see** moves without hammering refresh, you add something parallel—usually **WebSocket**, sometimes **SSE**—where the server can push “this operation landed” or a small **patch** after the fact.

I am completely fine shipping v1 on **HTTP only** with periodic refresh; realtime is as much a **product** bet as an engineering one, and pretending otherwise is how you inherit a socket layer you did not need yet.

### Read: board snapshot

Picture the first load of the board page. A natural first contract is:

`GET /boards/:boardId`

If your gateway prefixes everything with `/api`, that is fine—the **payload** is what you and the client will live with for months.

For **200**, the body should resemble what you already committed to in **Data modeling**. Returning a **nested** columns-with-cards snapshot early on is not a moral failure; it is often the fastest way to ship, as long as you know how you might evolve toward maps and ordered id lists later.

```json show
{
  "board": { "id": "board-1", "title": "Board 1", "version": 42 },
  "columns": [
    { "id": "col-1", "title": "Backlog", "order": 0 },
    { "id": "col-2", "title": "In Progress", "order": 1 },
    { "id": "col-3", "title": "Done", "order": 2 }
  ],
  "cards": [
    { "id": "card-1", "columnId": "col-1", "position": 0, "title": "Setup project structure" },
    { "id": "card-2", "columnId": "col-2", "position": 0, "title": "Install dependencies" },
    { "id": "card-3", "columnId": "col-1", "position": 1, "title": "Configure ESLint & Prettier" }
  ]
}
```

I would treat errors with the same care as loading and empty states in **Designing the UI**: decide what **timeouts**, **5xx**, and validation failures look like in **JSON**, not only as bare status codes, so the client can retry or explain instead of showing a generic “something broke.” A stable **`code`** is worth more than a clever English `message` you will regret string-matching in six months.

```json show
{
  "code": "BOARD_NOT_FOUND",
  "message": "Board does not exist or you do not have access."
}
```

### Pagination and large columns

If you are somewhere under “a few dozen to low hundreds” of cards, **one round trip** that pulls the whole board is often totally reasonable—I would not paginate for ceremony. Where life gets noisy is the **monster backlog**: years of cards, a single column that never shrinks, or many teams sharing one huge surface. Then you want enough to paint the screen quickly, and more as the user scrolls or digs into a lane.

**Cursor-based** pagination tends to survive reality better than naive page numbers when new items keep landing **at the top** of a column while someone is paging. You might extend the board read with something shaped like:

```json show
{
  "board": { "id": "board-1", "title": "Board 1", "version": 42 },
  "columns": [],
  "cards": [],
  "pageInfo": {
    "endCursor": "eyJjIjoiLi4uIn0",
    "hasNextPage": true
  }
}
```

When **one column** can hold **thousands** of cards, I would seriously consider splitting reads:

`GET /boards/:boardId/columns/:columnId/cards?cursor=&limit=`

—and then write down, in actual prose in your doc or spec, **how the cursor behaves** when inserts happen above the user’s window. That sounds pedantic until the frontend and backend quietly disagree about what “next page” means.

### Writes, idempotency, and retries

Reads are socially cheap; **writes** are where you discover you were never alone in the system—other users, and your own **double submits**, flaky Wi‑Fi, tab sleep, and all the ways a response can vanish after the server already committed. **Idempotency** is the promise that the **same logical action**, retried, does not become **two** irreversible effects. If I save a title twice, I still want one sensible outcome; if I delete a card someone else already removed, I want a **predictable** response—`200` with no-op, `204`, whatever you pick—more than I want a theatrical error.

A **resource-style** move is still a good teaching shape:

`PATCH /cards/:cardId`

```json show
{
  "columnId": "col-2",
  "afterCardId": null
}
```

It is easy to explain to product and easy to mock—but the failure mode that actually hurts is not the happy path; it is the request that **succeeded on the server** and whose response **never came back**, so you still want a **client-generated id** or an **`Idempotency-Key`** before you call that “done.”

### Moves as operations (recommended mental model)

On boards I usually push people past “PATCH three fields” and toward an **operation** the server can **validate**, **order**, and **dedupe**—because a move is an **intent**, and the world moves between your gesture and the commit.

```jsonc show
{
  "operationId": "uuid-client-generated",
  "type": "move_card",
  "cardId": "card-1",
  "fromColumnId": "col-1",
  "toColumnId": "col-2",
  "afterCardId": "card-2" // or toIndex: 3 — pick one convention and stick to it
}
```

**`operationId`** is the one you **reuse** when the user retries the **same gesture**—not a fresh UUID every time the transport blinks. **`afterCardId`** (or an index; pick **one** story and be honest about the trade-off) carries **intent**; the server is allowed to **adjust** if the neighbour card moved, and it should return enough **truth**—final order, **version**—that the client is not guessing in the dark.

`POST /boards/:boardId/operations` with `Idempotency-Key: <operationId>` (or uniqueness enforced on `operationId` in the body) is boring on purpose; boring is what saves you at 2 a.m.

On **success**, you hand back a bumped **`version`** (or **`seq`**) and either a **full snapshot**—blunt but clear—or a **minimal patch** (`{ affectedColumnIds, cards, order }`) if you are trying to be kind to bandwidth. **4xx** should carry something the UI can show; **409** should carry enough **server truth** that optimistic UI can roll back or reconcile without a mystery full refresh. Until the ack lands, the client is usually tracking **pending** somewhere—queue, flags, whatever matches your state library.

### Concurrency: version or sequence

Somewhere in the snapshot or the stream there has to be a number the client can compare: a **`version`** on the board (or per column), or a monotonic **`seq`** on applied operations if you are logging them that way. When paths cross, **server wins** plus **`409`** and a fresh fragment is the usual hammer for **whole-card moves**; **merge** becomes interesting when two people edit the **same card body** at once—that is a different problem than two drags racing.

Track **`lastSeenVersion`** or **`lastAppliedSeq`** on the client and learn to ignore **stale** noise.

### Realtime: event stream (optional)

When you add push, keep the payload small and the rules strict. Something like:

```json show
{
  "seq": 1001,
  "type": "move_applied",
  "payload": { "operationId": "…", "…final positions…" }
}
```

Order events by **`seq`**, reconnect with **`afterSeq`**, resync if you detect a gap, and treat duplicate **`operationId`** as ignorable on the client too—**idempotency** is not only a server problem.

### Hot paths

If you are only allowed to care about a few routes: **initial load** in one round trip (or paginated per column when you must), **move** as one mutation plus whatever broadcast you promised, and **in-board search**—if it exists—probably deserves its **own** read path instead of pretending the browser cache is a database.

### Auth

Thread **workspace / user** through the whole surface. On **403 vs 404** for “you cannot see this board,” pick a story that matches your privacy product and **stick to it**—consistency matters more than winning an argument on HTTP pedantry.

## Optimization strategies

By the time APIs are on the table, ordering is not an abstract exercise anymore—it shows up in **write amplification**, conflict behaviour, and how painful migrations become when you change your mind. That is why these next pieces sit here, right after **API design**, instead of being tucked away as a footnote.

### Ordering: index vs sort key

Two common patterns:

1. **Integer index** per column (`position: 0..n-1`). Simple to reason about; **every insert in the middle** can force **many rows** to update on the server unless you use fractional indexing or another trick.
2. **Lexicographic / fractional rank** string between neighbors (`rank` between `"a"` and `"b"`). Fewer row updates on insert; slightly more complexity in the client when computing “between.”

For learning, **explicit index + server normalizes** is enough; optimize when profiling shows write amplification.

### LexoRank: easy to hand-wave until it hurts

**LexoRank** is Atlassian’s open, battle-tested take on fractional / lexicographic ordering—Jira boards are the usual reference implementation people point to. Conceptually it is still “a string sort key between neighbors,” but the **details** (how you bucket, rebalance when strings get too long, and keep inserts cheap under concurrency) are where toy implementations diverge from production.

Teams often **start with integer `position`**, ship, and only later notice **write amplification** (many rows touched on one insert), **hot spots** in the index, or painful **migrations** when two clients resolve “move after card X” differently under load. LexoRank is the pattern many mature products adopt **before** those fires; it is also easy to **skip in interviews and greenfield code** because “just use an index” sounds sufficient until scale and collaboration show up.

For a walkthrough aimed at implementers, see [*Reordering Lists at Scale (Without Breaking Everything)*](https://www.youtube.com/watch?v=Z6c7txCqsu8) on YouTube. Atlassian publishes a reference implementation in the [**LexoRank** repo](https://bitbucket.org/atlassian/lexorank/src/main/). Roadmap: a focused **LexoRank for card ordering** pattern lives on the [Patterns](/patterns) index; this board guide stays the scenario-shaped home for *when* ordering strategy matters in a kanban model.

## Performance, reliability, and accessibility

- **Drag performance**: prefer **transform**-based dragging and avoid **layout thrash**; virtualize columns only when profiling proves you need it.
- **Realtime bursts**: **batch** UI updates per animation frame; dedupe by `operationId`.
- **Keyboard**: explicit **move to column** / **move up-down** commands; **roving tabindex** inside the board; **announce** column changes for screen readers.
- **Retries**: exponential backoff on **5xx** / network; never assign a **new** `operationId` for the **same user gesture**.

## Trade-offs (when you evolve past the first slice)

| Tension | One side | Other side | When the “other” wins |
| ------- | -------- | ---------- | --------------------- |
| Conflict handling | Server-normalized order after each move | Operational transform on shared fields | Multiple users edit **the same card body** concurrently |
| Transport | Poll + refetch | WebSocket push | Sub-second multi-user updates |
| Optimism | Wait for server | Optimistic + rollback | Drag-and-drop UX expectations |
| Column size | Load all cards | Paginate in column | Very large backlogs |

## Common failure modes

- **Duplicate cards** after a retry → **idempotency** on `operationId` + server dedupe.
- **Corrupt order** after tab sleep → **resync** from `version` / `seq`, not patch stacking forever.
- **Ghost moves** (UI shows move, server never got it) → **pending** timeout + user-visible retry or revert.
- **Accessibility gaps** → treat keyboard and SR as **v1**, not a later pass.
